COMPLAINT MANAGEMENT PORTAL (CMP) - FULL PROJECT FUNCTION OVERVIEW
Date: 2026-04-05

============================================================
0) QUICK ARCHITECTURE SUMMARY
============================================================
Tech stack:
  - Java Servlet/JSP web app (web.xml based, Servlet 2.5 style)
  - JDBC (MySQL)
  - NetBeans/Ant project layout (build.xml + nbproject)
  - Frontend: JSP + vanilla JS + app.css

Main folders:
  - src/java/com/cmp/...        Java source (Servlets, DAOs, Utils, Filter)
  - web/WEB-INF/views/*.jsp     JSP pages (all panels)
  - web/assets/css/app.css      UI styles
  - web/WEB-INF/web.xml         Routes + filter + listener

Request flow (high level):
  Browser -> (AuthFilter) -> Servlet (/login, /admin, /hod, /clerk, /user)
          -> DAO -> DB (JDBC via DBUtil)
          -> forward JSP via ViewUtil OR redirect based on role/action

============================================================
2) ROLE SYSTEM (WHO CAN ACCESS WHAT)
============================================================
Roles stored in `users.role`:
  - ADMIN, HOD, CLERK, STAFF, STUDENT, CR

Role routing after login:
  - ADMIN -> /admin
  - HOD   -> /hod
  - CLERK -> /clerk
  - else  -> /user  (covers STAFF/STUDENT/CR)

Authorization rules:
  - AuthFilter blocks all routes except /login, /index.jsp, /assets/* unless logged in.
  - Each servlet also enforces role checks:
      - AdminServlet  requires ADMIN
      - HodServlet    requires HOD
      - ClerkServlet  requires CLERK
      - UserServlet   requires logged-in (any role)

------------------------------------------------------------
3.1) LOGIN PANEL
View:    web/WEB-INF/views/login.jsp
Servlet: /login -> com.cmp.servlet.LoginServlet
DAO:     com.cmp.dao.AuthDao
------------------------------------------------------------
Purpose:
  - Authenticate using email + password.
  - Redirect user to role dashboard.
  - Force password change if `users.must_change_password = 1`.

UI:
  - Email + password form.
  - Password show/hide toggle.
  - Client-side email validation (format + normalize lowercase).

Backend functions used:
  - LoginServlet.doGet(): forward login.jsp
  - LoginServlet.doPost():
      - Validates email format
      - Calls AuthDao.authenticate(email, password)
      - On success: SessionUtil.login(...)
      - If mustChangePassword: redirect to `/user?action=changePasswordPage`
      - Else redirect based on role
  - AuthDao.authenticate(email, password):
      - Loads user by email where is_active = 1
      - Verifies password with PasswordUtil.matches(...)
      - Returns a Map (id, fullName, email, role, departmentId/name, classId/name, mustChangePassword)


------------------------------------------------------------
3.2) ADMIN PANEL
View:    web/WEB-INF/views/admin.jsp
Servlet: /admin -> com.cmp.servlet.AdminServlet
DAO:     com.cmp.dao.AdminDao
------------------------------------------------------------
Purpose:
  - Create/manage departments.
  - Create/manage HOD and Clerk accounts.
  - Assign HOD/Clerk to departments (via department update).
  - View system stats (department count, user counts, complaints stats).
  - Show “recent admin activities” notifications.

Main actions (POST /admin):
  - action=createDepartment
      Params: name, description, hodUserId (optional), clerkUserId (optional)
  - action=updateDepartment
      Params: departmentId, name, description, hodUserId, clerkUserId
  - action=deleteDepartment
      Params: departmentId
  - action=createRoleUser
      Params: fullName, email, role (HOD/CLERK), departmentId (optional)
      Result: Generates temp password, saves user, sends email.
  - action=updateRoleUser
      Params: userId, role, fullName, email
  - action=deleteRoleUser
      Params: userId, role
  - action=assignDepartment (present in servlet)
      Params: departmentId, hodUserId, clerkUserId
      Note: admin.jsp mostly uses updateDepartment for assignment; assignDepartment can be treated as an API.
  - action=clearDepartments / clearHods / clearClerks (present in servlet)
      Note: these actions exist server-side; admin.jsp may not render buttons for them.

Notifications:
  - GET /admin?action=markNotificationsRead returns 204 (client resets count locally).
  - AdminDao.getRecentAdminActivities() returns latest department/user activity rows.

Client-side validation:
  - admin.jsp includes JS validation for:
      - email format + duplicate email (HOD/CLERK)
      - department name uniqueness, etc.

3.3) HOD PANEL (DEPARTMENT ACADEMIC MANAGEMENT)
View:    web/WEB-INF/views/hod.jsp
Servlet: /hod -> com.cmp.servlet.HodServlet
DAO:     com.cmp.dao.HodDao
------------------------------------------------------------
Purpose:
  - Manage classes in the assigned department.
  - Manage student + staff accounts in the assigned department.
  - Bulk add students (Name,email list) into a class.
  - Assign/remove CR (class representative) role for students.
  - View complaint status summary counts for department.
  - Show “recent HOD activities” notifications.

Views inside HOD panel:
  - Uses query param `view` (default overview):
      - overview, classes, classStudents, staff, cr
  - Uses query param `classId` for classStudents view.

Main actions (POST /hod):
  - action=createClass
      Params: className, expectedStrength
  - action=updateClass
      Params: classId, className, expectedStrength
  - action=deleteClass
      Params: classId
  - action=createStudent
      Params: fullName, email, classId
      Result: creates account + temp password shown in UI
  - action=createBulkStudents
      Params: classId, studentEntries (multiline "Name,email")
      Result: creates multiple student accounts + returns `bulkCreatedUsers` list
  - action=updateStudent
      Params: studentId, fullName, email, classId
  - action=deleteStudent
      Params: studentId
  - action=createStaff
      Params: fullName, email
      Result: creates account + temp password shown in UI
  - action=updateStaff
      Params: staffId, fullName, email
  - action=deleteStaff
      Params: staffId
  - action=assignCr
      Params: studentId
      Rule: max 2 CR per class
  - action=removeCr
      Params: studentId
  - action=clearClasses / clearStudents / clearStaff (present in servlet)
      Note: server supports clearing; check UI before enabling (it’s destructive).

Notifications:
  - GET /hod?action=markNotificationsRead returns 204 (client resets count locally).
  - HodDao.getRecentHodActivities() returns latest class/user activity rows.

Client-side validation:
  - hod.jsp includes JS validation for:
      - email format and duplicate checks (based on current page data)
      - bulk list format and duplicates


------------------------------------------------------------
3.4) CLERK PANEL (COMPLAINT DESK)
View:    web/WEB-INF/views/clerk.jsp
Servlet: /clerk -> com.cmp.servlet.ClerkServlet
DAO:     com.cmp.dao.ClerkDao
------------------------------------------------------------
Purpose:
  - Show active complaints assigned to this clerk (non-SOLVED).
  - Update complaint status + remarks.
  - Show resolved history (SOLVED).
  - Notification badge based on `complaints.clerk_seen` flag.

Main actions (POST /clerk):
  - action=clearActive
      Deletes all complaints for this clerk where status <> 'SOLVED'
  - action=clearHistory
      Deletes all complaints for this clerk where status = 'SOLVED'
  - Default (no action set):
      Updates one complaint:
        Params: complaintId, status, remarks
        Status options used in UI: REJECTED, ASSIGNED, SOLVED, PENDING

Notifications:
  - GET /clerk?action=markNotificationsRead
      - Calls ClerkDao.markClerkSeen(clerkId)
      - returns 204

Important rule:
  - ClerkDao.updateStatus blocks updates if complaint is already SOLVED.
  - If status changes, it inserts a notification for the user (raised_by).

3.5) USER PANEL (STUDENT / STAFF / CR)
View:    web/WEB-INF/views/user.jsp
Servlet: /user -> com.cmp.servlet.UserServlet
DAO:     com.cmp.dao.UserDao
------------------------------------------------------------
Purpose:
  - Submit new complaints to department clerk.
  - See complaint history.
  - Edit complaints only when:
      - status is REJECTED, AND
      - within COMPLAINT_EDIT_WINDOW_MINUTES (default 30 minutes).
  - See notifications (from clerk status changes).
  - Mark notifications read.
  - Change password (first login flow).
  - Attachment streaming endpoints exist in UserServlet (see notes below).

Main actions (GET /user):
  - action=markNotificationsRead
      Marks notifications.is_read=1 for the user, returns 204.
  - action=changePasswordPage
      Forwards change-password.jsp.
  - action=downloadAttachment / viewAttachment
      Streams file from `/uploads` (force download vs inline view).
      Params: file (stored name), name (original name, optional)
  - No action:
      Loads complaints + notification counts and shows user.jsp
      Note: if role is ADMIN/HOD/CLERK it redirects to their dashboards instead.

Main actions (POST /user):
  - action=changePassword
      Params: newPassword
      Calls AuthDao.changePassword(userId, newPassword) and clears must_change_password.
  - action=createComplaint
      Params: scope (SELF / CLASS), title, description
      Notes:
        - CLASS scope only works if role is CR AND user has classId (server uses user.classId)
        - Duplicate complaint prevention (same user, same scope/title/desc, and not SOLVED)
        - Attachment saving is supported server-side (see notes below)
  - action=editComplaint
      Params: complaintId, title, description
      Rule:
        - only if raised_by matches
        - only if status is REJECTED
        - only if within edit window (minutes)
  - action=clearHistory
      Deletes all complaints raised by this user (server supports it)
      Note: user.jsp currently does not show a “clear history” button.

Attachment note (important):
  - UserServlet supports multipart upload under `Part attachment` and stores it under `/uploads`.
  - Current user.jsp does not include a file input or multipart form encoding, so attachment upload
    is effectively “backend-ready but UI-missing”.

5) DATABASE MODEL (INFERRED FROM QUERIES)
============================================================
This app assumes (at minimum) these tables:

users:
  - id (PK)
  - full_name
  - email (unique recommended)
  - password_hash
  - role  (ADMIN/HOD/CLERK/STAFF/STUDENT/CR)
  - department_id (FK departments.id, used by most roles)
  - class_id (FK classes.id, used by STUDENT/CR)
  - must_change_password (0/1)
  - is_active (0/1)
  - created_at (used by “recent activities” queries)

departments:
  - id (PK)
  - name
  - description
  - hod_user_id   (FK users.id)
  - clerk_user_id (FK users.id)

classes:
  - id (PK)
  - department_id (FK departments.id)
  - class_name
  - expected_strength

complaints:
  - id (PK)
  - raised_by (FK users.id)
  - department_id (FK departments.id)
  - class_id (nullable FK classes.id)
  - assigned_clerk_id (FK users.id)
  - complaint_scope (SELF / CLASS)
  - title
  - description
  - status (PENDING / ASSIGNED / REJECTED / SOLVED)
  - clerk_remarks
  - clerk_seen (0/1)
  - created_at, updated_at
  - attachment_name, attachment_path, attachment_type (optional columns)

notifications:
  - id (PK) (not explicitly queried, but typical)
  - user_id (FK users.id)
  - complaint_id (FK complaints.id)
  - message
  - is_read (0/1)
  - created_at

6) IMPORTANT BUSINESS RULES / CONSTRAINTS
============================================================
Email:
  - Normalized to lowercase + validated with regex server-side.
  - Uniqueness is enforced via DAO validation before insert/update.

Password:
  - Stored as SHA-256 hash (PasswordUtil).
  - New accounts get a generated temp password (email sent).
  - First login requires changePassword flow (must_change_password).

Complaint lifecycle:
  - Created as PENDING, assigned to department clerk.
  - Clerk can update status and add remarks.
  - When status changes, a notification is inserted for the user.

Complaint editing:
  - A user can edit only if:
      - status = REJECTED
      - within COMPLAINT_EDIT_WINDOW_MINUTES (default 30)

CR assignment:
  - Max 2 CR per class.

Class capacity:
  - Students+CR count cannot exceed classes.expected_strength.
  - Bulk add also validates remaining seats.
